# 06 权限系统

## AI Agent 的安全困境

2024 年以来，AI Agent 领域发生了多起引人关注的安全事件。有研究者演示了如何通过精心构造的 prompt injection，让 Agent 执行用户从未要求的危险命令。有人在公开的代码仓库中植入恶意注释，当 Agent 读取这些文件时会被诱导执行 `rm -rf /` 或者将敏感文件上传到外部服务器。这些事件揭示了一个根本性问题：**当你赋予 AI 执行真实操作的能力时，你同时也在赋予它造成真实损害的能力**。

这正是权限系统要解决的问题。它需要在两个矛盾的目标之间找到平衡：一方面让 Agent 足够自主，能够高效完成任务；另一方面限制它的行为边界，防止它做出不可逆的危险操作。

Claude Code 的权限系统是整个代码库中 **规模最大的单一模块**，超过 6,300 行代码分布在 25 个文件中。这个投入规模本身就说明了 Anthropic 对 Agent 安全性的重视程度。相比之下，很多开源 Agent 框架的权限机制只是一个简单的白名单/黑名单，甚至完全没有权限控制。

## 1️⃣ 三种权限模式

![三种权限模式](../imgs/06-perm-three-modes.png)


Claude Code 提供了三种权限模式，适应不同的使用场景和风险偏好：

| 模式 | 工作方式 | 适用场景 |
|------|---------|---------|
| **default** | 每次工具调用都弹窗让用户确认 | 最安全，适合接触敏感代码和生产环境时使用 |
| **auto** | ML 分类器预判风险，只有高风险操作才弹窗 | 日常开发，需要 Opus 4.6 或更高版本模型 |
| **plan** | 模型先规划动作并给出置信度评分，低置信度才弹窗 | 复杂的多步骤任务 |

### Default 模式：完全人工确认

这是最保守的模式。Agent 每次想要执行一个工具调用时，流程如下：

1. 向用户展示工具名称、参数、影响的文件路径
2. 用户选择：**Allow** 放行 / **Deny** 拒绝 / **Allow-All-For-This-Tool** 放行该工具的所有后续调用
3. 如果用户选择了 Allow-All，该工具在本次 session 内后续调用不再弹窗

Default 模式的优点是安全性最高，所有操作都经过人工审核。缺点是效率低下，一个复杂任务可能涉及几十次工具调用，每次都要确认非常打断工作流。这就是为什么 Claude Code 还提供了另外两种模式。

### Auto 模式：ML 分类器辅助判断

![YOLO分类器决策流程](../imgs/06-perm-yolo-classifier.png)


Auto 模式引入了一个 **ML 分类器** 来自动化安全判断，大幅减少需要人工确认的次数：

```
工具调用请求
  ↓
硬编码危险模式检查 → 命中 → 自动拒绝
  ↓ 未命中
YOLO 分类器预判
  ↓
高置信度安全 → 自动放行
低置信度或中风险 → 弹窗让用户确认
```

这里的 **YOLO 分类器** 名字很有意思。YOLO 在互联网文化中是 "You Only Live Once" 的缩写，暗示"放手去做"。但在 Claude Code 的上下文中，它指的是一个通过 side-query 调用 Claude 模型来评估命令安全性的分类器。它的实现在 `yoloClassifier.ts` 文件中，工作方式是：把即将执行的 bash 命令、命令模式、当前环境信息发送给 Claude，让模型判断这个命令是否安全，并返回匹配结果和置信度评分。

Auto 模式需要 Opus 4.6 或更高版本的模型支持，因为分类器的准确性直接依赖模型的判断能力。用较弱的模型运行分类器会导致误判率升高，要么放过了危险命令，要么频繁误报打断用户。

### Plan 模式：规划先行

Plan 模式的思路是让模型在执行之前先做规划：

1. 模型输出一份执行计划，列出即将执行的每一步
2. 每一步都带有一个 **置信度评分**，表示模型对该操作安全性的确信程度
3. 高置信度的步骤自动执行
4. 低置信度的步骤暂停，等待用户确认

这种模式特别适合复杂的多步骤任务。用户可以先看到完整的计划，对有疑虑的步骤进行干预，而不需要在每一步都做决定。它配合 `/plan` 命令使用，让用户可以主动触发规划模式。

## 2️⃣ 42 条危险模式：默认的安全防线

![42条危险模式](../imgs/06-perm-42-rules.png)


代码中硬编码了 **42 条自动拒绝的 bash 模式**。在默认权限配置下，这些模式会被自动拦截。

但需要澄清一个常见的误解：**这些规则并非绝对不可绕过。** 如果用户在 `settings.json` 中显式配置了 allow 规则，比如 `Bash(python*)`，那么 python 命令就会被放行。如果使用了 `bypassPermissions` 模式，所有限制都会被跳过。

这 42 条模式真正发挥作用的场景是：
- 在 **default 模式** 下，没有匹配到 allow 规则时，这些命令会弹窗要求用户确认
- 在 **auto 模式** 下，ML 分类器不会自动放行这些模式，必须走用户确认
- 进入 **auto 模式时**，Dangerous Rule Stripping 会自动剥离涉及这些模式的 allow 规则，防止分类器按旧规则自动放行

换句话说，这 42 条规则是 **auto 模式下的硬性安全底线**，在其他模式下则是一道需要用户主动确认才能通过的关卡。

这 42 条模式分为三类：

**跨平台代码执行，16 条**

```
python, node, deno, tsx, ruby, perl, php, lua,
npx, bunx, npm run, yarn run, pnpm run, bun run,
bash, sh
```

**Unix 专属，5 条**

```
zsh, fish, eval, exec, env/xargs/sudo
```

**内部专用，8 条，仅对 Anthropic 内部用户生效**

```
fa run, coo, gh/gh api, curl, wget,
git, kubectl, aws/gcloud/gsutil
```

为什么这些命令要被拦截？核心逻辑是：**任何能启动解释器或执行任意代码的命令都必须被拦截**。原因很直接：一旦 Agent 进入了 Python 解释器或 Node.js 运行时，它就可以在解释器内部执行任何操作，完全绕过 Claude Code 的所有其他权限检查。

举个例子，假设权限系统禁止直接执行 `rm -rf /`。如果 Agent 能够不受限制地执行 `python -c "import os; os.system('rm -rf /')"` ，那么对 `rm` 的限制就形同虚设。拦截解释器入口就是在堵住这个绕过路径。

这与操作系统安全设计中的 **最小权限原则** 一脉相承。Linux 的 SELinux 和 AppArmor 也是通过限制进程能够执行的系统调用来实现安全隔离。Claude Code 的危险模式检查可以类比为 Agent 层面的 syscall filter。

## 3️⃣ 权限规则配置

用户可以在 `~/.claude/settings.json` 中自定义权限规则，实现更精细的控制：

```json
{
  "permissions": {
    "allow": [
      "Bash(npm run build)",
      "Bash(git log *)",
      "Read",
      "Edit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force *)"
    ]
  }
}
```

规则支持 **通配符匹配**。`Bash(git log *)` 会匹配所有以 `git log` 开头的命令。规则的处理顺序是：

1. 检查 deny 规则，命中则自动拒绝
2. 检查 allow 规则，命中则自动放行
3. 都不命中，走当前权限模式的默认流程

这个设计让用户可以根据自己的工作场景做定制。比如一个前端开发者可以放行 `npm run build` 和 `npm run test`，因为这些命令在自己的项目中是安全的。但同一条命令在另一个项目中可能有完全不同的行为，所以这种配置天然是项目相关的。

## 4️⃣ 权限模式状态机

![权限模式状态机](../imgs/06-perm-state-machine.png)


三种模式之间的切换遵循严格的状态机规则：

```
default ──→ bypass，用户主动开启 dangerouslySkipPermissions
bypass  ──→ default，用户关闭或达到 N 次操作后自动回退
default ──→ auto，分类器启用且用户明确同意
```

这里有一个重要的安全机制：**Dangerous Rule Stripping**，危险规则剥离。

当用户从 default 模式切换到 auto 模式时，系统会自动检查用户之前配置的 allow 规则，把其中涉及危险模式的规则 **主动移除**。

为什么需要这个机制？考虑这样一个场景：用户在 default 模式下添加了 `Bash(python *)` 的 allow 规则，因为在 default 模式下每次调用都会弹窗确认，这条规则只是让确认少一步。但如果用户切换到 auto 模式，分类器可能会直接按照这条规则自动放行所有 python 命令。这就绕过了 42 条危险模式的硬编码保护。

Dangerous Rule Stripping 通过在模式切换时主动清理潜在的危险 allow 规则来防止这个漏洞。这是一个 **防御性编程** 的典型案例：不假设用户的配置在所有场景下都是安全的，而是在关键转换点做额外的安全检查。

## 5️⃣ 文件系统沙箱

![文件系统沙箱](../imgs/06-perm-sandbox.png)


除了命令级别的权限控制，Claude Code 还在文件系统层面做了隔离。`pathValidation.ts` 文件实现了路径校验逻辑：

- **工作目录限制**：所有文件操作被限制在当前工作目录及其子目录内。Agent 无法读写工作目录之外的文件
- **路径穿越防护**：禁止使用 `..` 进行路径穿越。即使 Agent 构造了 `../../etc/passwd` 这样的路径，也会被校验拦截
- **敏感目录保护**：写入 `.git`、`.claude` 等目录时会被拦截。这些目录包含版本控制信息和 Agent 配置，被篡改可能导致严重后果
- **额外工作目录**：可以通过配置添加额外的允许操作的目录，适应 monorepo 等需要跨目录访问的场景

这种沙箱设计借鉴了容器技术中的 **namespace 隔离** 思想。Docker 通过 mount namespace 限制容器能看到的文件系统范围，Claude Code 通过路径校验限制 Agent 能操作的文件范围，思路是一样的。

## 6️⃣ 行为学习：Denial Tracking

`denialTracking.ts` 实现了一个值得注意的功能：**用户拒绝行为追踪**。

系统会记录用户拒绝工具调用的次数和模式。如果同一类操作被反复拒绝，系统会调整后续的权限策略，倾向于对这类操作更加保守。

这个设计体现了一个重要的产品理念：权限系统应该随着使用而 **自适应**。如果用户总是拒绝某个操作，说明用户认为这个操作在当前场景下不合适，系统应该学会这个偏好，而不是每次都重复询问。

这类行为学习在传统安全系统中也有先例。防火墙的自适应规则、杀毒软件的启发式检测、手机 App 的权限推荐，都是基于用户行为模式来调整安全策略的例子。

## 7️⃣ 与业界方案的对比

| 项目 | 权限机制 | 安全层次 | 自适应能力 |
|------|---------|---------|-----------|
| **Claude Code** | 三模式 + 42 条硬编码规则 + ML 分类器 + 文件沙箱 + 行为学习 | 多层纵深防御 | 高，denial tracking + rule stripping |
| **Cursor** | 用户确认弹窗 | 单层 | 无 |
| **Aider** | 白名单命令 | 单层 | 无 |
| **OpenAI Codex CLI** | 沙箱模式 + 确认弹窗 | 两层 | 无 |
| **Devin** | 受限环境 + 人工审核 | 两层 | 有限 |

Claude Code 的权限系统在深度和广度上都远超同类产品。这种投入是有道理的：一个能够自主执行 shell 命令、读写任意文件的 Agent，其潜在风险远大于一个只做代码建议的工具。**权限系统是 Agent Harness 的核心**，模型能力再强，如果没有可靠的权限系统约束它的行为边界，就只是一个危险的玩具。Claude Code 在权限上投入了整个代码库中最多的工程资源，这个优先级排序值得所有做 Agent 的团队参考。
