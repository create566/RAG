# 11 AI Coding 时代的 Code Review

## 1️⃣ 从一次 Sourcemap 泄露说起

![Sourcemap泄露教训](../imgs/11-review-sourcemap-lesson.png)


2025 年，Claude Code 的完整源码被公开还原。原因令人哭笑不得：**构建流程中没有排除 sourcemap 文件**。Sourcemap 是 JavaScript 构建工具生成的调试辅助文件，记录了编译后代码与原始源码的映射关系。只要拿到 .map 文件，就能几乎完美地还原出原始代码。

这个项目有 51.5 万行代码，TypeScript 类型定义零错误，每个异步函数都有完善的错误处理，代码结构清晰整齐。从代码质量的角度看，它几乎无可挑剔。但就是在"构建产物应该包含哪些文件"这个最基础的发布安全问题上，出了致命的遗漏。

这个事件完美地揭示了 AI Coding 时代一个核心矛盾：**AI 在代码编写层面越来越强，但在工程化判断层面依然有盲区。** AI 能写出完美的代码，但它未必能判断这些代码的构建产物是否应该公开。

**这正是 Code Review 在 AI 时代需要重新定义的原因。**

## 2️⃣ 个人层面：Review 重心的转移

![Review重心转移](../imgs/11-review-focus-shift.png)


### 过去的做法

在 AI Coding 普及之前，Code Review 是软件工程中最重要的质量保障环节之一。Review 者会关注代码风格是否规范、变量命名是否清晰、设计模式是否合理、边界条件是否覆盖。一次认真的 CR 可能需要 30 分钟到数小时。

2024 年初，很多开发者刚开始使用 AI 写代码时，会对 AI 生成的每一行代码做仔细的 Review。这是一种谨慎的态度，在 AI 能力还不成熟的阶段完全合理。

### 现在的变化

随着模型能力的快速提升，这种逐行 Review 的方式正在变得不经济。**如果使用 AI 是为了提高生产效率，但 Review AI 代码的时间和自己写差不多，那 AI 的效率优势就被大幅削弱了。**

2025 到 2026 年，越来越多的工程师在实践中调整了自己的 CR 策略。核心趋势是：**把注意力从 AI 做得好的地方，转移到 AI 做不好的地方。**

AI 已经做得足够好的领域：

- **代码风格和命名规范。** AI 生成的代码通常比大多数人类工程师更一致、更规范。纠正 AI 的命名风格几乎是在浪费时间。
- **类型标注和接口定义。** 以 TypeScript 为例，AI 几乎不会漏掉类型标注，接口定义通常也很完整。
- **错误处理的覆盖度。** AI 倾向于为每个可能出错的地方都加上 try/catch 或错误检查。

AI 依然有盲区的领域，也是人类 Review 应该聚焦的地方：

- **业务逻辑的正确性。** AI 按照 prompt 写代码，但 prompt 本身可能有歧义，或者遗漏了某些业务规则。逻辑对不对，只有理解业务的人才能判断。
- **安全性。** AI 不会主动考虑注入攻击、权限越界、信息泄露等安全问题。它能写出功能正确的代码，但未必是安全的代码。
- **架构决策。** 某个模块该不该拆分、某个依赖该不该引入、某个接口该不该对外暴露。AI 给出的是局部最优解，全局的架构判断需要人来做。

**CR 的目标正在从"找代码问题"变成"验证 AI 的判断是否符合预期"。**

## 3️⃣ 团队层面：共享 System Prompt 基准

一个开发团队如果每个成员都用自己的 prompt 来指导 AI 写代码，最终合并到一起的代码库将会风格混乱、架构不一致。每个人的 AI 可能用不同的错误处理方式、不同的命名约定、不同的测试策略。

解决这个问题的方法是建立一份**团队共享的 System Prompt 基准**。在 Claude Code 中，这就是 CLAUDE.md 文件。在其他 AI 编码工具中，也有等效的配置机制。

这份共享 prompt 应该明确定义：

**技术栈和架构约定。** 项目使用什么框架、什么语言版本、什么构建工具。模块之间的依赖关系是怎样的。数据流是怎么走的。

**代码风格和命名规范。** 变量用 camelCase 还是 snake_case，文件名用什么格式，组件怎么组织。

**提交规范和分支策略。** commit message 的格式，分支命名规则，PR 的合并策略。

**测试要求。** 哪些代码必须有单元测试，覆盖率的最低要求是多少，集成测试怎么写。

**安全红线。** 哪些操作绝对不能出现在代码中，比如硬编码密钥、明文存储密码、不受限的文件访问。

这和传统团队中的 Coding Standard 文档本质上是同一回事。区别在于：过去的受众是人类工程师，现在的受众变成了 AI。而 AI 比人类更擅长严格遵守规范，前提是规范被清晰地表达在 prompt 中。

**没有共享 prompt 的团队，每个人的 AI 写出来的代码风格、架构风格、测试风格都不一样。** 代码库最终会变成一锅大杂烩。有了共享 prompt，至少保证了一个一致的基准线。

## 4️⃣ CICD 基础设施层面：AI 时代的新范式

![AI时代CICD新检查项](../imgs/11-review-cicd-new.png)


这是最需要被重新定义的一层，也是最有价值的一层。

### 传统 CICD 的通过标准

传统的 CICD 流水线关注的是：代码能不能编译通过、能不能正常运行、测试是否全部通过。这些检查假设代码是由有经验的工程师手动编写的，问题主要出在逻辑错误和疏忽。

### AI 时代 CICD 需要新增的检查项

当代码的主要生产者从人类变成 AI，CICD 流水线需要增加一系列针对 AI 代码特征的检查。

**1️⃣ 敏感信息扫描**

AI 生成代码时，可能会把 prompt 中提到的 API Key、内部 URL、数据库连接字符串等敏感信息硬编码到代码中。CI 流水线应该自动扫描这些模式，任何硬编码的凭据都应该触发构建失败。

常用的工具包括 git-secrets、truffleHog、gitleaks。这些工具在 AI 时代之前就存在，但现在变得更加必要。

**2️⃣ 隐私检查**

AI 有时候会把 prompt 中的信息泄露到代码注释、日志输出、错误消息中。比如你在 prompt 中描述了一个内部系统的架构，AI 可能把这段描述作为注释写进了代码里。CI 流水线应该扫描代码中是否出现了不该出现的内部信息。

**3️⃣ 构建产物审计**

Claude Code 的 sourcemap 泄露就是这个检查项缺失导致的。CI 流水线应该检查最终的构建产物中是否包含：

- .map 文件，也就是 sourcemap
- debug 符号表
- 内部文档或注释
- 测试数据和 fixture
- 环境配置文件

**4️⃣ 依赖安全审计**

AI 倾向于引入第三方库来解决问题。每个新的依赖都是一个潜在的供应链攻击面。CI 流水线必须对依赖做安全审计。

在 JavaScript/Node.js 生态中，`npm audit` 是最基本的工具。更专业的工具如 Snyk、Socket.dev 可以做更深层次的分析，包括检测恶意包、许可证合规性等。

**5️⃣ Code Review Agent**

这是最有前景的方向：**引入一个 Review Agent，对每一次提交做自动化审查。**

具体做法是把需求 prompt 和代码实现一起提交。Review Agent 读取 prompt，看代码实现是否符合预期。这实现了一种**AI 检查 AI** 的闭环。

```
提交内容：
├── prompt.md              # 原始需求 prompt
├── src/feature.ts         # AI 生成的代码
└── tests/feature.test.ts  # AI 生成的测试

Review Agent 的检查清单：
1. 代码实现是否符合 prompt.md 中描述的需求？
2. 有没有安全漏洞？注入、越权、信息泄露
3. 有没有硬编码的敏感信息？
4. 测试覆盖了哪些场景？遗漏了哪些边界条件？
5. 有没有引入不必要的依赖？
6. 代码中是否存在 prompt 信息泄露？
```

**AI 时代的 CICD 新范式应该被正式确立，作为 Harness Engineering 的一部分，甚至是最重要的一部分。** 当代码的生产者从人变成了 AI，传统的人工 Review 已经不够用了。你需要用基础设施来系统性地保证质量。

## 5️⃣ 一套可落地的 AI 代码 Review 流程

![五阶段Review流程](../imgs/11-review-five-stages.png)


把上述所有层面整合起来，形成一套完整的流程：

```
第一步：AI 生成代码
  附带原始 prompt 一起提交
     ↓
第二步：自动化检查，CI 流水线
  - lint / type check / build
  - 敏感信息扫描
  - 构建产物审计，sourcemap / debug 文件检测
  - 依赖安全审计
     ↓
第三步：Review Agent，AI 审查 AI
  - 需求符合度检查
  - 安全审计
  - 边界条件覆盖
  - prompt 信息泄露检查
     ↓
第四步：人工 Review，只看关键路径
  - 业务逻辑正确性
  - 架构决策合理性
  - 安全攻击面评估
     ↓
第五步：合并发布
```

这套流程的核心思想是**分层过滤**。自动化检查过滤掉最基础的问题，Review Agent 过滤掉 AI 特有的问题，人工 Review 只需要关注最高层级的判断。每一层都做自己最擅长的事情。

## 6️⃣ 从 Claude Code 源码学到的 Review 教训

![AI代码特征与Review重点](../imgs/11-review-ai-traits.png)


Claude Code 51.5 万行源码是一个绝佳的 AI 代码 Review 案例。通过分析它的代码特征，可以提炼出一套针对 AI 代码的 Review 指南：

| AI 代码的典型特征 | 应该重点检查什么 |
|-----------------|----------------|
| 每个 async 函数都有 try/catch | catch 块里是否只做了 log 就把错误吞掉了？被吞掉的错误会导致问题被隐藏，排查时极其困难 |
| 类型标注极其完整 | 有没有用 `as any` 强行绕过类型系统？`as any` 是类型安全的逃生舱，每一个都值得审视 |
| 所有分支条件都有处理 | else 分支是否只是一个占位符，里面什么都没做？AI 喜欢写"完整"的代码，但 else 里可能只是 return undefined |
| 错误消息写得很完整 | 错误消息中是否泄露了内部实现细节？比如包含了文件路径、数据库表名、内部 API 地址 |
| 代码量大但结构整齐 | 是否存在大量重复代码可以抽象？AI 倾向于复制粘贴相似的逻辑，而非提取公共函数 |
| 依赖引入很多 | 是否有不必要的第三方库？AI 可能为了一个简单的功能引入一整个库 |
| 构建通过，测试全绿 | 构建产物中是否包含不该出现的文件？这正是 Claude Code sourcemap 泄露的教训 |

## 7️⃣ 核心原则

**AI 写代码的效率优势在于生成速度。** 它能在几秒钟内写出几百行功能完整的代码，这是人类做不到的。

**人工的不可替代价值在于安全判断和工程化决策。** AI 看到的是当前任务，人看到的是整个系统。AI 优化的是功能正确性，人需要把控的是安全性、可维护性、架构合理性。

**CICD 基础设施是把这两者串联起来的 Harness。** 它把人类的工程化经验编码成自动化规则，持续不断地应用到每一次代码提交上。

在 AI Coding 时代，**最危险的不是 AI 写出了有 bug 的代码，而是团队在 AI 代码上放松了工程化标准。** 代码的生产效率提高了十倍，质量保障的投入也应该相应提高，方式则需要从人力密集型转向基础设施密集型。
