# 04 上下文工程

## 1️⃣ 什么是上下文工程，为什么它是 Agent 的命门

2025 年，Shopify CEO Tobi Lütke 在社交媒体上提出了一个观点：我们应该停止说 **prompt engineering**，改说 **context engineering**。这个说法迅速在技术社区传播开来，因为它精准地抓住了 AI 应用开发中一个被长期低估的核心挑战。

要理解上下文工程，先要理解大语言模型的一个基本限制：**上下文窗口是有限的。**

上下文窗口可以理解为模型的 **工作记忆**。人类在思考一个复杂问题时，大脑能同时保持关注的信息量是有限的，大语言模型也是如此。Claude 不同版本的上下文窗口从 200K 到 1M token 不等，Opus 4.6 已经支持 1M token。听起来很大，但在一个复杂的 Agent 任务中，这个空间很快就会被消耗殆尽。

想象一下 Claude Code 在处理一个大型代码重构任务时需要装进上下文的东西：系统指令、用户的个性化配置、项目规则、当前 Git 仓库的状态、几十轮对话的完整历史、每次工具调用的结果、40 多个工具的定义文档。所有这些信息都在争夺同一块有限的空间。

上下文工程要解决的核心问题就是：**如何在有限的空间里塞进最有价值的信息，同时保证模型的注意力不被无关内容稀释。**

这和图书馆管理有异曲同工之妙。一个好的图书馆员不会把所有的书都堆在读者的桌上，而是根据读者当前的需求，精准地取来最相关的几本，并且按照一定的顺序摆放。上下文工程做的就是这件事：帮模型 **筛选信息、排列优先级、动态调整**。

在 Claude Code 的源码中，上下文工程的实现散布在多个文件中，但核心思想可以归纳为一套 **分层优先级 + 预取缓存 + 动态注入** 的完整体系。

## 2️⃣ System Prompt 构建：模型的操作手册

![System Prompt 构建](../imgs/04-ctx-system-prompt.png)

System prompt 是每次 API 调用都会携带的基础指令，你可以把它理解为 **模型的操作手册**。Claude Code 的 system prompt 不是一段静态文本，而是通过两个阶段动态构建出来的。

### 静态部分：底层规则

第一阶段加载的是不会随会话变化的基础信息：

- **内置 Agent 行为规则**：告诉模型它是一个编程助手，应该如何与用户交互，什么操作需要请求许可，什么情况下应该主动提供更多信息
- **可用的 slash command 列表**：模型需要知道用户可以使用哪些快捷命令
- **模型配置和 thinking 参数**：Extended Thinking 是否启用，推理深度设置
- **权限模式描述**：当前处于哪种权限模式，哪些操作需要用户确认

### 动态部分：实时情报

第二阶段加载的信息会根据当前环境实时变化，这是上下文工程最精细的部分。

**Git 状态注入** 的实现在 context.ts 文件中。系统会并行执行 4 个 git 命令来收集信息：当前所在分支、`git status --short` 输出的文件变更列表、最近一次 commit 的信息、当前用户名。这四个命令的输出被拼接成一段结构化文本，注入到 system prompt 中。

这里有几个值得注意的工程细节。首先，Git 状态的输出被 **严格限制在 2000 字符以内**。如果一个大型项目有几百个变更文件，status 的输出可能长达数千字符，全部塞进 system prompt 是一种浪费。超出部分会被截断，并附带一条警告告诉模型信息不完整。其次，整个 Git 状态在 session 内只获取一次，后续通过 **memoize 缓存** 直接复用。最后，在远程模式或用户明确禁用的情况下，这一步会被完全跳过。

**CLAUDE.md 记忆系统** 是 Claude Code 上下文工程中最复杂的模块，实现代码在 claudemd.ts 中，长达 **1,400 行**。

CLAUDE.md 是一种让用户告诉 Claude 项目规则和个人偏好的机制。但 Claude Code 的设计远超一个简单的配置文件。它建立了一套 **五层优先级的层级加载系统**，从低到高分别是：

![CLAUDE.md 五层加载](../imgs/04-ctx-five-layers.png)

第一层是 **全局管理级**，路径为 `/etc/claude-code/CLAUDE.md`。这个位置通常由企业 IT 管理员配置，用于在整个组织范围内强制执行统一的规则。比如"所有代码变更必须包含测试"或者"禁止直接操作生产数据库"。

第二层是 **用户级**，路径为 `~/.claude/CLAUDE.md`。这是用户个人的全局配置，适用于所有项目。比如"我偏好 TypeScript"或者"提交信息用中文"。

第三层是 **项目级**，包括项目根目录的 `CLAUDE.md`、`.claude/CLAUDE.md`、以及 `.claude/rules/` 目录下的所有 `.md` 文件。这一层定义项目特定的规则，通常被提交到 Git 仓库中，团队共享。

第四层是 **本地级**，文件名为 `CLAUDE.local.md`。这个文件被 gitignore，专门存放个人的私有配置。比如本地的 API 密钥路径或者特殊的开发环境设置。

第五层是 **自动记忆**，通过 TEAMMEM 这个 feature flag 控制。这是一个实验性功能，系统会自动从对话中提取有价值的信息并持久化。

这五层的加载顺序意味着高优先级的配置会覆盖低优先级的配置。一个项目级的规则可以覆盖用户级的默认设置，而本地级的配置又可以覆盖项目级的规则。

CLAUDE.md 还支持一个强大的 **@include 指令系统**，允许一个 CLAUDE.md 文件引用其他文件的内容。语法支持四种路径格式：`@path` 相对路径、`@./relative` 显式相对路径、`@~/home` 用户目录路径、`@/absolute` 绝对路径。系统只在 **叶子文本节点** 中解析 @include 指令，不会在代码块里误触发。同时有 **循环引用检测** 机制，通过一个 Set 数据结构追踪已经处理过的文件路径，防止 A 引用 B、B 又引用 A 导致的无限递归。系统支持 200 多种文件扩展名，会自动屏蔽二进制文件。

整个记忆系统有几个 **关键常量** 控制其边界：

- `MAX_MEMORY_CHARACTER_COUNT = 40000`：所有 CLAUDE.md 内容合并后的总字符上限
- `MAX_ENTRYPOINT_LINES = 200`：MEMORY.md 文件的最大行数
- `MAX_ENTRYPOINT_BYTES = 25000`：MEMORY.md 文件的最大字节数

这些限制确保记忆系统不会过度膨胀，在上下文窗口中占据过多空间。

**权限规则** 从两个位置加载：`~/.claude/settings.json` 是用户级设置，`.claude-permissions.json` 是项目级设置。两者合并后解析为 **ToolPermissionContext** 数据结构，包含 allow、deny、ask 三种规则。这些规则会被注入到 system prompt 中，让模型知道哪些操作可以直接执行、哪些需要用户确认、哪些被明确禁止。

## 3️⃣ 分层优先级：当空间不够时，谁先走

![分层压缩优先级](../imgs/04-ctx-compression-priority.png)

上下文工程最关键的设计决策之一是 **当空间不够用时，应该保留什么、丢弃什么**。Claude Code 建立了一套清晰的六级优先级体系：

```
优先级从高到低：

1. 系统指令         ← 永远保留，绝不压缩
2. CLAUDE.md 用户记忆 ← 永远保留，绝不压缩
3. 权限规则         ← 永远保留，绝不压缩
4. 最近的对话       ← 最后才被压缩
5. 历史工具调用结果  ← 最先被微压缩清理
6. 早期对话历史     ← Full Compact 时被摘要替换
```

这个优先级设计背后有深刻的产品逻辑。

**前三级永远不被压缩。** 这意味着无论 Agent 循环了多少轮、无论上下文被压缩了多少次，用户的个性化配置、项目规则和权限设置始终完整保留。想象一下如果权限规则在某次压缩中被意外丢弃了，Agent 可能会执行本该被禁止的危险操作。再想象一下如果 CLAUDE.md 中"提交信息用中文"的规则被丢弃了，Agent 突然切换到英文提交。这些行为不一致会严重损害用户信任。

**第四级是最近的对话。** 这是模型理解当前任务的直接依据，压缩它会导致模型失去对当前工作进度的感知。所以它是最后才会被动手的部分。

**第五级和第六级是最先被清理的。** 早期的工具调用结果和对话历史，随着时间推移，它们的信息价值在递减。一个小时前读取的文件内容，很可能已经不再相关。把它们替换为简洁的摘要，可以释放大量空间给更有价值的新信息。

这套优先级体系有一个优雅的特性：**用户无需了解它的存在就能受益**。用户不需要知道什么是上下文窗口、什么是 token 限制、什么是压缩策略。他们只需要正常使用 Claude Code，系统会在后台自动管理一切。但如果有一天用户发现 Agent 在第 30 轮循环后依然记得自己在 CLAUDE.md 中写的项目规范，那种"它真的懂我"的感觉，正是这套上下文工程的功劳。

## 4️⃣ 预取与缓存：为速度而战

Agent 循环的每一轮都需要读取上下文信息。如果每一轮都从头开始做 I/O 操作，性能会非常糟糕。Claude Code 为此建立了一套 **预取 + 缓存** 体系：

| 信息类型 | 预取策略 | 缓存策略 |
|---------|---------|---------|
| Git 状态 | query loop 开始时异步预取 | session 内 memoize，只取一次 |
| CLAUDE.md | 应用启动时异步加载 | session 内缓存，不重新读取 |
| Skill 列表 | 后台 discovery 进程 | 文件监听器，变化时热更新 |
| 工具定义 | 启动时一次性注册 | feature flag 控制，运行时不变 |

**预取** 的核心思想是 **让 I/O 提前于需求发生**。在 query loop 真正需要 Git 状态信息之前，预取操作就已经在后台启动了。等到构建上下文的阶段需要用到这个信息时，数据已经准备好了，零等待。

**缓存** 的核心思想是 **避免重复劳动**。CLAUDE.md 的内容在一次会话中几乎不会变化，读一次就够了。Git 状态在大部分场景下也是稳定的。通过 memoize 缓存，后续的循环直接使用上一次的结果。

Skill 列表的缓存策略比较特殊。它使用 **文件监听器** 而不是简单的 memoize。因为用户可能在 Agent 运行过程中添加新的 Skill 文件，系统需要感知到这个变化并热更新。但它不会在每次循环都重新扫描文件系统，只有在监听器检测到变化事件时才触发更新。

这套预取和缓存体系在数字上的影响是显著的。一个 20 轮的 Agent 循环，如果每轮都做完整的 I/O 操作，假设每次读取 CLAUDE.md、Git 状态、Skill 列表各需要 30 毫秒，20 轮就是 1.8 秒的纯等待时间。通过预取和缓存，这个时间可以降低到接近零。

## 5️⃣ 动态上下文注入：在正确的时间提供正确的信息

上下文工程的另一个核心原则是 **不要一股脑把所有信息都塞给模型**。

这个原则违反了很多人的直觉。人们往往认为给模型的信息越多越好。但研究和实践都表明，过量的上下文信息会 **稀释模型的注意力**，导致它在关键信息上的处理能力下降。学术界把这种现象叫做 **Lost in the Middle**，意思是当上下文很长时，模型对中间部分信息的关注度会显著降低。

![Lost in the Middle 现象](../imgs/04-ctx-lost-in-middle.png)

Claude Code 的解决方案是 **每轮循环都重新组装上下文，根据当前状态决定注入什么**。

具体来说，上下文的组装不是一次性完成后就不变的。每一轮 Agent 循环的「构建上下文」阶段，系统都会重新评估当前状态，决定带入哪些信息：

**对话初期**，模型还没有开始调用工具，system prompt 中的 Skill 列表、项目结构信息、CLAUDE.md 规则等全局信息占比很高，给模型提供全局视野来规划接下来该做什么。

**多轮执行后**，消息历史中已经积累了大量的工具调用结果（文件内容、命令输出等），这些具体的执行上下文自然地成为模型注意力的焦点。system prompt 中的全局信息虽然还在，但模型的注意力已经聚焦到消息历史中与当前任务直接相关的内容上。

**压缩触发后**，旧的消息历史被替换成摘要，同时 transcript 文件路径被注入。模型在压缩后的第一轮循环中，可以通过摘要快速重建对之前工作的理解，如果需要某个被压缩掉的细节，可以主动调用文件读取工具去 transcript 里按需获取。

需要澄清的是，这里没有一个显式的「阶段切换」逻辑。源码中不存在 `if (phase === 'planning')` 这样的分支判断。真正发生的是：system prompt 每轮都重新构建，压缩系统在必要时介入清理，而模型自己决定什么时候规划、什么时候执行。系统做的是 **持续地为模型准备最相关的上下文**，而不是硬编码不同阶段的注入策略。

这种设计的价值在于它 **把上下文管理变成了一个持续的过程**。每一个循环迭代，系统都在根据当前状态重新评估：什么信息现在最重要？什么信息可以先放一放？什么信息应该被完全移除？

## 6️⃣ Prompt Cache 优化：节省真金白银

![Prompt Cache 优化](../imgs/04-ctx-prompt-cache.png)

Anthropic 的 API 有一个 **Prompt Cache** 机制。如果两次 API 调用的请求前缀相同，第二次调用可以复用第一次的计算缓存，显著减少延迟和费用。对于 Agent 这种需要频繁调用 API 的场景，Prompt Cache 的命中率直接影响运营成本。

Claude Code 对 Prompt Cache 做了两个层面的精细管理。

**第一个层面是 CacheSafeParams。** 这是一组在父子 Agent 之间共享的参数，包括 systemPrompt、userContext、systemContext、toolUseContext 四个字段。系统确保这四个字段在主 Agent 和所有 fork 出来的子 Agent 之间保持完全一致。这样，子 Agent 的 API 调用就能命中父 Agent 已经建立的 Prompt Cache。

这个设计的经济价值非常直观。假设 system prompt 有 10,000 个 token。如果没有 CacheSafeParams，每次 fork 一个子 Agent 都要重新处理这 10,000 个 token。在一个复杂任务中可能 fork 十几个子 Agent，这就是 15 万个 token 的额外开销。有了 CacheSafeParams，这些全部可以命中缓存，节省的费用可能高达数美元。

**第二个层面是 cache break 检测。** 系统专门设置了一个监控模块，持续追踪 Prompt Cache 的命中率。如果命中率突然下降，系统会分析原因：是代码 bug 导致的真正 cache break，还是压缩操作或微压缩导致的 **预期内下降**。

这里有一个来自 Anthropic 内部的生产教训。2026 年 3 月 22 日的一次事后分析发现，**77% 的工具相关 cache break** 都是同一个原因造成的：AgentTool 和 SkillTool 的 description 字段里嵌入了 **动态列表**。每次可用的 Agent 或 Skill 列表发生变化，工具定义的文本就会改变，导致 system prompt 的哈希值变化，Prompt Cache 随之失效。

这个案例生动地说明了上下文工程的难度。一个看似合理的设计——"在工具描述中列出所有可用的子工具"——在 Prompt Cache 的视角下却是一个性能杀手。修复方案是把动态列表从工具描述中移除，放到会话级别的上下文中，让工具定义本身保持稳定。

## 7️⃣ 与其他系统的上下文管理对比

不同的 Agent 系统对上下文管理的投入程度差异巨大。

**LangChain 的 Memory 模块** 提供了几种基础的上下文管理策略：ConversationBufferMemory 存储完整历史、ConversationSummaryMemory 保留摘要、ConversationBufferWindowMemory 只保留最近 N 轮。这些策略简单直观，但都是 **全局一刀切** 的。它没有分层优先级，没有预取缓存，没有动态注入。当上下文接近上限时，要么全部保留要么粗暴截断。

**OpenAI Assistants API** 通过 Threads 管理对话历史，提供了一定程度的自动上下文管理。但它是一个黑盒，开发者无法控制压缩策略、无法设置优先级、无法做精细的缓存优化。

**Cursor** 据报道在上下文管理上做了大量工作，特别是在代码上下文的精准检索方面。它使用了 **向量检索** 技术来找到与当前任务最相关的代码片段。这和 Claude Code 的方法互补：一个侧重于 **检索精度**，一个侧重于 **全生命周期管理**。

**Claude Code** 的上下文工程在行业内处于 **系统性最强** 的位置。五层优先级的记忆加载、六级的压缩优先级、预取缓存体系、动态注入策略、Prompt Cache 精细管理，形成了一套完整的端到端解决方案。它的投入程度体现在 1,400 行的 CLAUDE.md 加载代码上，也体现在专门为 cache break 设置检测模块的工程纪律上。

## 8️⃣ 总结

上下文工程是 Agent 产品用户体验的 **隐形支柱**。用户看不到它，但它决定了 Agent 是否能在第 30 轮循环后依然记得项目规则，是否能在压缩历史信息后依然理解当前任务，是否能在 fork 子 Agent 时保持成本可控。

Claude Code 在这个领域的投入强度，从代码量和设计精细度上看，在行业内都属于 **顶级水平**。它证明了一个观点：在 AI Agent 领域，**管理好模型能看到的信息，和模型本身的能力一样重要。**
