# 05 消息压缩系统

## 为什么 AI Agent 都绕不开压缩问题

每一个大语言模型都有一个硬性限制：**上下文窗口**。你可以把它理解为模型的"工作记忆容量"，就像人类在解题时脑子里能同时装下的信息量一样，模型在一次对话中能处理的文本总量是有上限的。Claude 的上下文窗口是 200K token，GPT-4o 是 128K token，这些数字看起来很大，但在 Agent 场景下消耗速度极快。

为什么消耗这么快？因为 Agent 和普通聊天不同。普通聊天是人和模型一问一答，每条消息几十到几百个 token。但 Agent 需要读文件、执行命令、搜索代码，每一次工具调用的返回结果可能就是几千个 token。一个 `FileRead` 读一个 500 行的源代码文件，轻松吃掉 2000-3000 token。一个复杂的编程任务可能涉及 **几十次工具调用**，每次都在往上下文里塞入大量内容。按这个速度，十几轮交互就能把 200K 的窗口填满。

业界的其他 Agent 项目也面临同样的问题。**Cursor** 的做法是把长对话拆分成多个独立请求，每次只发送最相关的上下文片段。**Aider** 采用了 repo map 的方式，维护一份代码库的结构化摘要，避免把完整文件内容塞进上下文。**Devin** 在早期版本中甚至直接截断旧消息，导致 Agent 经常"失忆"，忘记之前做过什么。

Claude Code 的解决方案是目前业界最精细的：**一个三层递进的压缩体系**。它的代码位于 `src/services/compact/` 目录，包含 26 个文件，是整个 Claude Code 代码库中工程化程度最高的模块之一。

## 1️⃣ 三层压缩体系全景

![三层压缩体系全景](../imgs/05-compact-three-layers.png)

在深入每一层之前，先看整体架构：

```
每次 API 调用前
  ↓
第一层：微压缩 Microcompact
  - 清理过时的工具调用结果
  - 每次调用都执行，无需阈值
  ↓
检查 token 用量是否超过安全线
  ↓ 超过
第二层：Session Memory Compact
  - 把关键信息沉淀到记忆文件
  - 从消息历史中删除已沉淀的旧消息
  ↓ 空间仍然不够
第三层：Full Compact
  - 启动独立 Agent 做全量摘要
  - 九段式结构化模板
  - 用一条摘要消息替换全部历史
```

这三层的设计哲学是 **渐进式降级**。第一层成本最低，每次都做；第二层中等成本，只在接近上限时做；第三层成本最高，是最后的兜底手段。这种分层策略避免了"要么不压缩，要么一次性全压缩"的极端做法，在信息保留和空间释放之间找到了平衡。

## 2️⃣ 第一层：微压缩

![微压缩机制](../imgs/05-compact-microcompact.png)


### 核心思想

微压缩的目标很简单：**清理那些已经不再有用的工具调用结果**。

想象你让 Agent 先读了一个文件，然后修改了它，再读了修改后的版本。第一次读取的结果在修改之后就过时了，继续保留在上下文里只是浪费空间。微压缩就是自动清理这类过时内容。

### 处理范围

微压缩只处理特定工具的输出，这些工具的共同特点是 **产出量大但时效性差**：

- **FileRead、FileEdit、FileWrite**：文件操作，内容可能已经被后续编辑覆盖
- **Bash 及所有 shell 工具**：命令执行结果，通常只在执行当时有参考价值
- **Grep、Glob**：搜索结果，随着文件修改可能已经不准确
- **WebSearch、WebFetch**：网页内容，信息已被 Agent 消化

有几类内容 **永远不会被微压缩触碰**：用户消息、assistant 的文本回复、thinking block。这些内容要么是用户的原始意图表达，要么是模型的推理过程，丢了就无法还原。

### 两种实现方式

Claude Code 实现了两种微压缩机制，它们解决的是同一个问题，但技术路径不同。

**时间触发型微压缩**

这是比较直接的方式。当距离上一次 assistant 消息超过一定时间，说明 Anthropic API 服务端的 **prompt cache** 已经过期。prompt cache 是 API 层的一个性能优化：如果你连续多次请求发送了大量相同的前缀内容，服务端会缓存这部分内容，避免重复处理，从而降低延迟和费用。但这个缓存有时效性，过期之后那些旧内容反正也要重新处理，不如直接清掉。

具体做法是把旧的 tool_result 内容替换为 `[Old tool result content cleared]`，只保留最近 N 个工具结果不动。这样模型知道"这里之前有过一次工具调用，但内容已被清理"，不会因为突然缺少消息而困惑。

```
触发条件：距上次 assistant 消息的分钟数 > gapThresholdMinutes
保留策略：保留最近 keepRecent 个结果，最少保留 1 个
实际操作：tool_result.content = "[Old tool result content cleared]"
```

**缓存编辑型微压缩**

这是一种更精巧的方式，通过 feature flag `CACHED_MICROCOMPACT` 控制。它不修改本地的消息内容，而是通过 API 层的 `cache_edits` 机制，告诉服务端"请删除指定 tool_result 的缓存"。

这种方式的优势在于 **不破坏已有的 prompt cache**。时间触发型需要修改消息内容，修改之后整个 prompt 的 hash 值变了，所有缓存都失效。缓存编辑型只是告诉服务端"这一块不要了"，其余部分的缓存仍然有效，性能更好。

```
触发条件：已注册的 tool_result 数量 > triggerThreshold
保留策略：保留最近 keepRecent 个
实际操作：生成 cache_edits block 发给 API，本地消息不变
```

两种方式 **互斥**，缓存编辑型优先。如果模型不支持 cache_edits，或者是外部构建的版本，就自动降级到时间触发型。

## 3️⃣ 第二层：Session Memory Compact

### 动机

当微压缩不足以控制上下文增长，系统需要更激进的手段。但直接做全量摘要成本很高，需要启动一个独立的 LLM 调用，而且摘要过程本身会丢失细节。有没有折中方案？

Session Memory Compact 就是这个折中方案。它的思路类似于人类做笔记：**把重要的东西记到笔记本上，然后把脑子里的原始信息清掉**。

### 工作流程

当自动压缩触发时，系统 **优先尝试** Session Memory Compact：

1. 扫描消息历史，识别出关键信息：重要的决策、发现的问题、用户的偏好、文件修改记录等
2. 把这些信息写入持久化的 session memory 文件
3. 从消息历史中删除已经被记录的旧消息
4. 检查释放的空间是否足够

如果 Session Memory 能释放足够空间，压缩到此结束，不需要进入代价更高的 Full Compact。

### 和 Full Compact 的区别

Session Memory Compact 保留的是 **结构化的关键信息**，而 Full Compact 生成的是 **叙述性的摘要文本**。前者更精确但覆盖面有限，后者覆盖面广但会丢失细节。两者配合使用，形成"先精准沉淀，不够再全面摘要"的策略。

## 4️⃣ 第三层：Full Compact

![九段式压缩模板](../imgs/05-compact-nine-segments.png)


### 触发条件

当 token 用量超过 **上下文窗口减去 13,000 token 的缓冲区**，且 Session Memory Compact 释放的空间不够时，Full Compact 启动。

关键常量如下：

```
AUTOCOMPACT_BUFFER_TOKENS = 13,000       // 触发缓冲区
WARNING_THRESHOLD_BUFFER_TOKENS = 20,000 // 警告阈值
MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20,000   // 摘要最大输出
MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3 // 熔断器阈值
```

13,000 token 的缓冲区设计是为了给模型留出足够的 **生成空间**。如果等到上下文完全填满才压缩，模型连回复的空间都没有了。20,000 token 的警告阈值则是一个更早的预警线，提醒系统"快要满了"。

### 独立 Agent 执行

Full Compact 的一个重要设计决策是：**压缩工作由一个独立 fork 出来的 Agent 完成**，不在主循环里做。

为什么？因为压缩本身就是一次 LLM 调用，如果在主循环里做，用户需要等待压缩完成才能继续交互。fork 出独立 Agent 后，主循环可以先给用户反馈"正在压缩上下文"，然后异步等待结果。

这个独立的压缩 Agent 收到整个对话历史，按照 **九段式模板** 生成结构化摘要：

```
1. 用户的主要请求和意图
2. 关键技术概念
3. 涉及的文件和代码片段，带完整代码
4. 遇到的错误和修复方式
5. 问题解决过程
6. 所有用户消息原文，排除工具结果
7. 待完成的任务
8. 当前正在进行的工作
9. 下一步计划
```

这个模板的设计非常讲究。第 1-5 段是 **回顾性的**，记录"发生了什么"；第 6 段 **保留用户原话**，这极其重要，因为用户的原始表述往往包含摘要无法捕捉的细微意图；第 7-9 段是 **前瞻性的**，确保压缩后 Agent 知道接下来该做什么。

很多其他 Agent 的压缩策略只做回顾性摘要，压缩后 Agent 就"忘了"自己正在做什么，需要用户重新提醒。Claude Code 的九段式模板通过显式包含"当前工作"和"下一步计划"来解决这个问题。

### analysis + summary 两阶段生成

压缩 prompt 要求 LLM 先在 `<analysis>` 标签里做分析思考，再在 `<summary>` 标签里写正式摘要。最终只保留 summary 部分注入上下文，analysis 被丢弃。

这是一个经典的 **Chain-of-Thought** 应用。让模型先"想清楚"哪些信息重要、哪些可以丢弃，然后再动手写摘要。分析阶段可能会产生几千 token 的推理过程，但这些推理不需要保留在最终的上下文里，只是一个提升摘要质量的中间步骤。

### 两种压缩方向

- **全量压缩 BASE**：摘要整个对话历史，生成一条全新的摘要消息替换所有旧消息
- **部分压缩 PARTIAL**：只摘要一部分消息。分为 `from` 方向和 `up_to` 方向，前者是"从某条消息开始往后摘要"，后者是"从头摘要到某条消息为止"。部分压缩会保留未被摘要的消息原文

部分压缩的典型场景是：对话前半部分已经压缩过一次了，后半部分是新增的内容。此时只需要摘要后半部分，和前半部分的摘要拼接即可。

### 压缩后处理

压缩完成后有一系列收尾工作：

1. **消息替换**：旧消息全部删除，替换为一条包含摘要的 user message
2. **Transcript 备份**：完整的原始对话写入 transcript 文件，这是不可逆操作的安全网
3. **状态重置**：微压缩状态、Session Memory 指针、上下文折叠状态全部重置
4. **Cache 通知**：通知 prompt cache detection 模块"这次 cache 命中率下降是压缩导致的，不要报假警"

## 5️⃣ 工程化细节

![熔断器机制](../imgs/05-compact-circuit-breaker.png)


### 熔断器

如果自动压缩连续失败 3 次，**停止重试**。这个看似简单的机制背后有一段真实的生产事故：

代码注释记录了 2026 年 3 月 10 日发现的问题：1,279 个 session 出现了 50 次以上的连续压缩失败，最严重的一个 session 失败了 **3,272 次**。每次失败都会触发一次 API 调用去重试，整个系统每天因此浪费约 **25 万次 API 调用**。加了熔断器之后这个问题彻底消失了。

熔断器是微服务架构中的经典模式。Netflix 在 2012 年开源了 Hystrix 熔断器库，此后这个模式被广泛应用于分布式系统中。但在 LLM Agent 的上下文管理领域看到熔断器应用，说明这个领域的工程成熟度已经开始向传统后端系统看齐。

### 递归保护

压缩本身是通过 LLM 调用完成的。如果压缩 Agent 自己的上下文也超了怎么办？会不会触发对压缩 Agent 的压缩，形成无限递归？

代码里做了显式的递归守护：

```typescript
if (querySource === 'session_memory' || querySource === 'compact') {
  return false  // 不再触发自动压缩
}
```

同理，**Context Collapse** 模式下也不触发自动压缩。Context Collapse 是另一种上下文管理策略，它和自动压缩会互相竞争资源，同时运行会导致不可预测的行为。

### Prompt Cache 感知

压缩会破坏已有的 prompt cache，因为消息内容变了，缓存的 hash 值自然对不上。代码里有专门的 `notifyCompaction` 机制，在压缩发生时通知 cache break detection 模块。

这个机制解决的是 **可观测性问题**。Anthropic 内部有监控系统追踪 prompt cache 命中率，如果命中率突然下降通常意味着某个地方出了 bug。但压缩导致的命中率下降是正常的、预期内的。如果不做区分，运维团队会被大量假警报淹没。

代码注释记录了 2026 年 3 月 1 日的一次生产教训：Session Memory Compact 后缺少 cache baseline 重置，导致 **20% 的 cache break 事件是误报**。

### 与业界方案的对比

| 项目 | 压缩策略 | 信息保留度 | 工程复杂度 |
|------|---------|-----------|-----------|
| **Claude Code** | 三层递进，微压缩 + Session Memory + Full Compact | 高，九段式模板保留关键信息 | 26 个文件，高度工程化 |
| **Cursor** | 对话分割，每次只发送相关片段 | 中，依赖相关性判断 | 中等 |
| **Aider** | Repo Map 结构化摘要 | 中，只保留结构信息 | 较低 |
| **OpenAI Codex CLI** | 简单截断 + 摘要 | 低，早期消息直接丢失 | 较低 |

Claude Code 的方案在信息保留度和工程复杂度上都是最高的，这与它作为长时间自主运行 Agent 的定位一致。一个需要在单个 session 中完成复杂编程任务的 Agent，对上下文管理的要求远高于一个只做代码补全的工具。
